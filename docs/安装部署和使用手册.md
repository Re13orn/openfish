# OpenFish（小鱼）：安装部署和使用手册

本文档面向本项目当前实现（单用户、本机运行、Telegram 长轮询、Codex CLI 执行引擎）。

## 1. 功能概览

你可以通过 Telegram 远程执行以下能力：

- 启动引导：`/start`
- 项目管理：`/projects`、`/use <project>`、`/project-root [abs_path]`、`/project-add <key> [abs_path] [name]`
- 发起任务：`/do <task>`、`/ask <question>`
- 模板任务：`/templates`、`/run <template> [附加说明]`
- Skills 管理：`/skills`、`/skill-install <source>`
- 连续性控制：`/status`、`/last`、`/retry [附加说明]`、`/resume`、`/cancel`
- 审批流：`/approve [note]`、`/reject [reason]`
- 项目记忆：`/memory`、`/note <text>`
- 变更查看：`/diff`、上传策略：`/upload_policy`
- 帮助：`/help`
- 文件分析：直接发送文档（会下载到当前项目临时目录后进行只读分析）
- 便捷提问：直接发送普通文本（等效于 `/ask`）

当前 Telegram 交互不是“只能记命令”的纯命令行模式，而是混合模式：

- 主键盘：`项目`、`提问`、`执行`、`状态`、`继续`、`变更`、`定时`、`更多`、`帮助`
- 高频控制入口会显示为可复用卡片，优先更新已有消息，而不是不断刷新消息
- 高摩擦操作优先走按钮和分步向导，不要求你每次都手打完整命令
- 审批按钮、审批备注、拒绝原因都绑定显式 `approval_id`，避免旧按钮误触

## 2. 前置条件

请先准备：

1. Python 3.12+
2. 已安装并可用的 Codex CLI（`codex` 命令可执行）
3. Telegram Bot Token（通过 BotFather 创建）
4. 你的 Telegram 用户 ID（加入白名单）
5. 本机可访问的代码仓库目录

建议在 Linux/macOS 上部署，Windows 可用 WSL2。

## 3. 目录说明

仓库关键路径：

- 根目录配置样例：`env.example`、`projects.example.yaml`、`schema.sql`
- 应用目录：`mvp_scaffold/`
- 应用源码：`mvp_scaffold/src/`
- 运行数据：`mvp_scaffold/data/`

## 4. 安装步骤（开发/单机运行）

以下命令从仓库根目录执行。

### 4.1 推荐路径：4 步跑通

项目已提供一键脚本：`mvp_scaffold/scripts/install_start.sh`

```bash
# 1) 安装依赖并初始化本地文件
bash mvp_scaffold/scripts/install_start.sh install

# 2) 最小化配置向导
bash mvp_scaffold/scripts/install_start.sh configure

# 3) 首次自检
bash mvp_scaffold/scripts/install_start.sh check

# 4) 后台启动
bash mvp_scaffold/scripts/install_start.sh start
```

`configure` 现在只要求填写这些关键项：

- Telegram Bot Token
- Telegram 用户 ID
- 默认项目根目录
- 第一个项目目录

脚本会自动：

- 写入 `.env`
- 写入最小化 `mvp_scaffold/projects.yaml`
- 保留其余运行参数为默认值

如果你还不知道自己的 Telegram 用户 ID：

1. 先给 bot 私聊一条消息，例如 `/start`
2. 再执行 `bash mvp_scaffold/scripts/install_start.sh tg-user-id`

### 4.2 常用脚本命令

```bash
bash mvp_scaffold/scripts/install_start.sh configure
bash mvp_scaffold/scripts/install_start.sh check
bash mvp_scaffold/scripts/install_start.sh tg-user-id
bash mvp_scaffold/scripts/install_start.sh tg-user-id your_telegram_username
bash mvp_scaffold/scripts/install_start.sh run
bash mvp_scaffold/scripts/install_start.sh stop
bash mvp_scaffold/scripts/install_start.sh restart
bash mvp_scaffold/scripts/install_start.sh status
bash mvp_scaffold/scripts/install_start.sh logs
```

说明：

- `install` 默认会询问是否立即进入最小化配置向导。
- 非交互场景（CI）可用 `SKIP_CONFIG_WIZARD=1` 跳过向导：
  `SKIP_CONFIG_WIZARD=1 bash mvp_scaffold/scripts/install_start.sh install`
- `check` 会校验 `.env`、`projects.yaml`、Codex CLI、Telegram API 连通性以及项目路径。
- `start` 会以当前配置启动单实例服务；重复启动会被进程锁拦下。

### 4.3 手动安装（高级用户）

```bash
cd mvp_scaffold
python3.12 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

### 4.4 手动准备项目注册表

```bash
cp ../projects.example.yaml ./projects.yaml
```

然后编辑 `mvp_scaffold/projects.yaml`，至少保留 1 个真实项目，重点检查：

- `path` 指向真实存在目录
- `allowed_directories` 覆盖允许访问范围
- `test_command`、`default_branch` 可按需填写
- 可选设置 `default_project_root`，用于 `/project-add <key>` 时自动创建 `<default_project_root>/<key>`

### 4.5 手动准备环境变量

可在仓库根目录创建 `.env`：

```bash
cd ..
cp env.example .env
```

编辑 `.env` 最少需要改这些字段：

- `TELEGRAM_BOT_TOKEN`
- `ALLOWED_TELEGRAM_USER_IDS`
- `PROJECTS_CONFIG_PATH=./mvp_scaffold/projects.yaml`
- `SQLITE_PATH=./mvp_scaffold/data/app.db`
- `MIGRATIONS_DIR=./mvp_scaffold/migrations`
- `CODEX_BIN=codex`

加载环境变量：

```bash
set -a
source .env
set +a
```

注意：当前实现不自动读取 `.env` 文件，必须由 shell 显式 `source` 或用系统服务注入环境变量。

### 4.6 手动首次启动

```bash
cd mvp_scaffold
source .venv/bin/activate
python -m src.main
```

启动后会自动：

- 读取项目注册表 YAML
- 初始化 SQLite（按 `schema.sql` + `migrations/`）
- 开始 Telegram 长轮询

## 5. 首次验收（建议）

先执行：

```bash
bash mvp_scaffold/scripts/install_start.sh check
```

自检通过后启动服务，再在 Telegram 私聊你的 bot，按顺序执行：

1. `/start`
2. 点主键盘里的“项目”，确认项目面板能正常弹出
3. 点选一个项目，或手动执行 `/use <你的项目key>`
4. 点主键盘里的“状态”，确认状态卡片正常返回
5. 点主键盘里的“提问”，输入：`这个项目是做什么的`
6. 点主键盘里的“执行”，输入：`仅分析当前项目目录结构并给出摘要，不修改文件`
7. 点主键盘里的“更多”，确认能看到 `审批面板`、`MCP`、`Skills`、`上传策略`
8. 发送一个小文件（如 `README.md`）测试上传分析

如出现审批提示，再测试：

1. 直接点 `批准` / `拒绝`
2. 再试 `批准+备注` / `拒绝+原因`
3. 确认审批卡片和状态卡片不会无限新增重复消息

## 6. 生产常驻部署

### 6.1 Linux（systemd）

新建 `/etc/systemd/system/openfish-assistant.service`：

```ini
[Unit]
Description=OpenFish Assistant
After=network.target

[Service]
Type=simple
User=YOUR_USER
WorkingDirectory=/ABS/PATH/codex-remote/mvp_scaffold
EnvironmentFile=/ABS/PATH/codex-remote/.env
ExecStart=/ABS/PATH/codex-remote/mvp_scaffold/.venv/bin/python -m src.main
Restart=on-failure
RestartSec=3

[Install]
WantedBy=multi-user.target
```

启用与查看日志：

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now openfish-assistant
sudo systemctl status openfish-assistant
journalctl -u openfish-assistant -f
```

### 6.2 macOS（launchd）

新建 `~/Library/LaunchAgents/com.local.openfish.assistant.plist`：

## 7. 日常使用说明

### 7.1 常用流程

1. 用主键盘“项目”打开项目面板，优先直接点最近项目
2. 用“提问”做只读分析，用“执行”发起改动类任务
3. 用“状态”查看当前项目、最近摘要、审批状态、下一步
4. 用“继续”恢复最近任务，用“变更”查看工作区摘要
5. 用“定时”查看或创建周期任务
6. 用“更多”进入低频入口：`审批面板`、`模板`、`Skills`、`MCP`、`上传策略`
7. 需要精确操作时，仍可直接手打命令：`/projects`、`/do`、`/ask`、`/status` 等

### 7.2 当前交互模型

- 无参高频操作：优先按钮点击
- 有参高频操作：优先进入向导
- 纯文本：在已选项目下等效 `/ask`
- 审批：优先按钮处理，备注/原因走两步向导
- 面板类结果：`状态`、`项目`、`定时`、`审批`、`更多` 会优先更新最近卡片
- 如果你更偏命令式，也可以继续直接使用完整命令

### 7.3 命令建议

- `/ask` 里写“只读，不改文件”可进一步保守
- `/do` 尽量写清验收标准（例如“只改最小范围，并运行 XXX 测试”）
- 高风险任务建议先 `/ask` 再 `/do`
- 上传文件仅允许白名单后缀且受大小限制（见 `.env` 上传相关配置）
- 默认上传大小上限为 `200MB`（`MAX_UPLOAD_SIZE_BYTES=209715200`）

## 8. 数据与日志

默认数据位置（可被环境变量覆盖）：

- SQLite：`mvp_scaffold/data/app.db`
- 审计日志：`audit_logs` 表
- 任务与审批：`tasks`、`task_events`、`approvals` 表
- 项目状态/记忆：`project_state`、`project_memory` 表

可快速查看最近任务：

```bash
sqlite3 mvp_scaffold/data/app.db "select id,command_type,status,created_at,latest_summary from tasks order by id desc limit 10;"
```

## 9. 升级与回滚

升级建议：

1. 停服务
2. 备份 `mvp_scaffold/data/app.db`
3. 更新代码与依赖
4. 启动服务（自动执行 schema 初始化和迁移）
5. 执行 `/status` 验证

回滚建议：

1. 停服务
2. 恢复代码版本
3. 必要时恢复 SQLite 备份
4. 重启并检查 `/status`

## 10. 常见问题排查

1. 提示“未授权用户”
原因：`ALLOWED_TELEGRAM_USER_IDS` 未包含你的 Telegram ID。

2. 提示“未选择活跃项目”
先执行：`/use <project>`

3. 提示“项目路径不存在”
检查 `projects.yaml` 的 `path` 是否真实存在、挂载是否正常。

4. 提示 “Codex binary not found”
确认 `codex` 在服务环境 PATH 中，或把 `CODEX_BIN` 改为绝对路径。

5. bot 无响应
检查：
- 进程是否在运行
- token 是否正确
- 本机网络是否可访问 Telegram API
- 日志是否出现异常堆栈

6. 一直等待审批
先看“状态”卡片或打开“审批面板”，优先直接点按钮；
如果需要补备注或原因，再走 `批准+备注` / `拒绝+原因`。

7. 面板消息很多，聊天太乱
当前实现会优先更新最近的 `状态`、`项目`、`定时`、`审批`、`更多` 卡片。
如果你看到旧卡片没有更新，通常是因为消息已过旧，重新点一次对应主键盘入口即可。

## 11. 安全建议

- 仅把你的个人 Telegram ID 加入白名单
- `allowed_directories` 只配置必要路径，避免放宽到 Home 根目录
- 审批提示不明确时优先 `/reject`，再改写更精确任务
- 定期备份 `app.db`

---

如需后续扩展（例如 Webhook、多用户、Web UI），建议另开分支，不要破坏当前单用户本机模型。
