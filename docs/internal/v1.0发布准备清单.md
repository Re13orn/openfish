# OpenFish v1.0 发布准备清单

这个清单只回答一个问题：什么时候 `v0.9.x` 可以升级成 `v1.0.0`。

## 1. 版本语义先定清楚

`v1.0.0` 对 OpenFish 的含义建议固定为：

1. 产品边界稳定
- 单用户
- Telegram 驱动
- 本地 Codex 助手

2. 核心交互稳定
- 主菜单
- 面板
- 按钮优先
- 向导
- 状态卡片

3. 核心命令语义稳定
- `v1.x` 内不轻易做破坏性改名或重定义

4. 首次安装路径稳定
- `install`
- `configure`
- `check`
- `start`

## 2. P0：发 `v1.0` 前必须完成

1. 清理 `CHANGELOG.md`
- `Unreleased` 不能继续堆历史版本内容
- `v1.0.0` 条目必须完整

2. 跑完整人工 smoke test
- 使用 [SMOKE_TEST_CHECKLIST.md](/Users/apple/workspace/Dev/AI/OpenFish/docs/internal/SMOKE_TEST_CHECKLIST.md)

3. README 与手册同步
- `README.md`
- `README_CN.md`
- `docs/安装部署和使用手册.md`
- `docs/ARCHITECTURE.md`

4. 明确已知限制
- 单用户
- 本地环境依赖 `codex`
- MCP 依赖本地配置与本地运行时
- 不是托管服务

5. 准备正式 release notes
- `docs/releases/v1.0.0.md`

## 3. P1：强烈建议在 `v1.0` 前完成

1. 增加基础 `/usage` 能力
- 至少能看当前会话或当前项目的基础使用信息

2. 补一轮 FAQ / 故障恢复说明
- Telegram 网络问题
- MCP 配置问题
- `codex` 不可用问题
- 首次配置失败问题

3. 再做一轮人工文案收敛
- 保证 TG、README、手册里的术语一致

## 4. 不需要阻塞 `v1.0` 的事情

下面这些可以放到 `v1.1+`：

1. 更细的 stream 过程流
2. 更完整的 usage/cost 统计
3. 更丰富的 MCP 可视化
4. 更多 skill / template 生态

## 5. 推荐发布节奏

最稳妥的路线：

1. 再发几个 `v0.9.x`
- 只修 bug 和文档问题

2. 发 `v1.0.0-rc1`
- 冻结功能面
- 只允许 bugfix

3. RC 稳定后再发 `v1.0.0`

## 6. `v1.0` 通过标准

只有当下面条件全部满足，才建议发 `v1.0.0`：

1. 全量测试通过
2. 人工 smoke test 通过
3. 文档一致
4. `CHANGELOG` 整理完成
5. release notes 就绪
6. 没有已知的严重交互阻塞问题
