# OpenFish（小鱼）：产品介绍演讲稿（含使用方法）

> 适用场景：团队分享、Demo Day、内部立项汇报  
> 建议时长：8-12 分钟  
> 讲述方式：一边讲，一边现场在 Telegram 演示

---

## 0. 开场（30 秒）

大家好，今天我介绍的是 **OpenFish（小鱼）**。  
它解决一个非常实际的问题：**离开电脑后，如何继续安全地推进本地项目开发任务**。

一句话定义：  
**这是一个运行在你自己电脑上的、单用户、Telegram 驱动、Codex 执行的远程开发助手。**

---

## 1. 我们要解决的痛点（1 分钟）

很多开发者都有类似经历：

- 在路上、会议间隙、非工作位时，临时想确认项目状态
- 想让 AI 帮忙分析项目问题，但不希望把代码和权限放到云端
- 手机上直接看终端日志体验很差，信息噪音很大

所以我们的目标不是“做一个万能 Agent 平台”，而是做一个：

- 单人可控
- 本地执行
- 项目连续
- 手机友好
- 默认保守

的实用工具。

---

## 2. 产品定位与设计理念（1.5 分钟）

### 2.1 产品定位

这个系统是：

- 单用户
- 本地优先
- Telegram 远程入口
- Codex CLI 执行引擎
- SQLite + YAML 持久化

这个系统不是：

- 多租户平台
- 公网远程 Shell
- 企业级编排系统
- 插件市场

### 2.2 设计理念

1. **Single-user first**：只服务一个可信 owner  
2. **Local-first**：所有执行与状态都留在本机  
3. **Project-centric**：按项目维护连续性，不靠聊天记录“猜上下文”  
4. **Conservative by default**：白名单、目录约束、审批流、安全上传  
5. **Small & understandable**：单进程、模块清晰、便于维护

---

## 3. 核心架构（1 分钟）

系统是单进程模块化架构：

- `telegram_adapter`：消息收发与长轮询
- `command_router`：命令分发与流程编排
- `project_registry`：YAML 项目注册
- `task_store`：SQLite 任务与状态持久化
- `codex_runner`：封装 Codex CLI
- `approval_service`：审批状态管理
- `audit_logger`：关键行为审计

好处是：**结构简单但可持续迭代**，调试和回溯成本低。

---

## 4. 功能介绍（2 分钟）

### 4.1 项目与任务

- `/projects` 查看项目
- `/use <project>` 选择当前项目
- `/ask <question>` 只读分析
- `/do <task>` 执行任务

### 4.2 连续性与效率

- `/status` 查看当前项目、会话、最近摘要、审批状态
- `/last` 查看最近任务详情
- `/retry [附加说明]` 一键重跑最近 ask/do 任务
- `/resume` 继续未完成工作流

### 4.3 审批与安全

- `/approve`、`/reject`
- 任务可进入 `waiting_approval`，显式确认后继续
- 全程审计，状态可追溯

### 4.4 文件上传分析（安全版）

- Telegram 文档上传后下载到当前项目临时目录
- 后缀白名单校验（已支持 `.apk`）
- 文件大小限制（默认 200MB）
- 路径合法性、敏感命名、符号链接风险校验
- 再交给 Codex 做分析

## 5. 如何使用（可直接照读+照做）（2 分钟）

### 5.1 一次性安装

```bash
bash mvp_scaffold/scripts/install_start.sh install
bash mvp_scaffold/scripts/install_start.sh configure
```

配置向导会引导填写：

- `TELEGRAM_BOT_TOKEN`
- `ALLOWED_TELEGRAM_USER_IDS`
- `projects.yaml` 项目信息
- Codex 与上传策略参数

可用辅助命令自动获取 Telegram 数字 ID：

```bash
bash mvp_scaffold/scripts/install_start.sh tg-user-id
```

### 5.2 启动与运维

```bash
bash mvp_scaffold/scripts/install_start.sh start
bash mvp_scaffold/scripts/install_start.sh status
bash mvp_scaffold/scripts/install_start.sh logs
```

### 5.3 Telegram 日常操作

建议首轮顺序：

1. `/help`
2. `/projects`
3. `/use demo`
4. `/status`
5. `/ask 这个项目是做什么的`
6. `/do 请检查当前项目风险并给出下一步计划`
7. `/last`
8. `/retry 补充要求：只读，不改文件`

上传文件分析：直接给 bot 发文档即可。

## 6. 现场演示脚本（主持人提词版）

> 你可以在演讲时按下面台词讲 + 同步操作手机

**台词 A：选择项目**
- “我先看有哪些项目：`/projects`”
- “切换到 demo：`/use demo`”
- “看状态：`/status`”

**台词 B：发起任务**
- “我先做只读分析：`/ask 这个仓库的核心模块是什么`”
- “然后发执行任务：`/do 帮我梳理一版最小改动方案`”

**台词 C：连续性与效率**
- “我随时可以看最近任务：`/last`”
- “如果结果不错但想加约束，直接重跑：`/retry 补充要求：不要修改代码`”

**台词 D：安全上传**
- “现在我上传一个文件，系统会先做白名单和大小校验，再分析”
- “这保证了它不是一个无限制远程文件入口”

**台词 E：收尾**
- “整个流程没有离开本机执行，所有状态和审计都在本地可回溯”

## 7. 开发思路与过程复盘（1 分钟）

我们采用“分阶段交付”的方式：

- **Phase 1**：先打通可运行闭环（Telegram + 项目 + SQLite + Codex）
- **Phase 2**：做连续性与易用性（status/resume/last/retry/templates）
- **Phase 3**：强化审批流与记忆模型（approve/reject/memory/note）

工程上坚持三点：

1. 小模块明确职责  
2. 每次改动都可测试、可回滚  
3. 优先做能长期稳定跑的能力，而不是一次性堆功能

---

## 8. 安全与可靠性亮点（30 秒）

- Telegram 用户白名单鉴权
- 任务必须绑定注册项目
- 严格目录约束与路径校验
- 上传文件安全策略（后缀/大小/路径/命名）
- 审批状态显式化
- 全链路审计日志
- Telegram 网络抖动下的发送重试与降级

---

## 9. 结语（30 秒）

这套系统的价值不在“炫技”，而在**稳定、可控、可持续**：  
它让你在任何地方，都能以手机为入口，继续推进你自己的本地项目，同时保持安全边界和工程秩序。

如果要总结成一句话：  
**不是把开发搬到云上，而是把你的本地开发能力安全地延伸到手机上。**
